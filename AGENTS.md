# AGENTS.md

## Project overview

TEEBUSINESS is a server-rendered Django storefront for footwear/streetwear. It covers catalog and stock management, a session cart and guest checkout, customer accounts with OTP verification, newsletters/product alerts, printable invoices, and a customized Django Unfold admin.

There is no project README. Treat this file and the code as the source of truth; the historical React/Vite `dist/` build was removed and there is no separate frontend or API architecture.

## Tech stack

- Python 3 / Django 5.2 (`requirements.txt` pins Django `<6.0`)
- SQLite with a custom UUID-based `accounts.User`
- Django templates, Tailwind utility classes, small amounts of custom CSS, and Alpine.js
- Tailwind v4 is compiled locally; Alpine remains loaded from jsDelivr in the storefront base template
- WhiteNoise for production static files
- Django Unfold and django-import-export for admin UX and CSV/XLSX/JSON exports
- ReportLab for admin PDF exports; printable order invoices are HTML
- Brevo HTTPS API, SMTP, or console email behind a logging/retry backend
- Pillow is used for product image optimization, but is not currently declared in `requirements.txt`
- Twilio and Celery/Redis are optional and not active dependencies

No formatter, linter, type checker, CI workflow, or automated deployment script was found. Node is used only for the Tailwind build workflow.

## Repository layout

- `teebusiness_core/`: settings, root URLs, WSGI/ASGI, shared admin PDF export
- `shop/`: catalog, products, sizes/stock, images, currency display, catalog seeding
- `cart/`: session cart, checkout, persisted orders/items, invoice views, admin dashboard metrics
- `accounts/`: custom user, email login, OTP flows, lockout/security logs, profile and password flows
- `communication/`: subscribers, campaigns, email history/backends, product-alert signals
- `pages/`: home and informational/contact pages
- `templates/`: active project-level storefront, account, cart, email, include, and admin templates
- `frontend/`: Tailwind v4 source, explicit scan paths, dynamic-class safelist, and build notes
- `<app>/templates/`: older or fallback templates; project-level `templates/` wins for duplicate paths
- `static/`: source assets that must stay tracked; `staticfiles/` is generated by `collectstatic`
- `media/products/`: uploaded product images
- `locale/{fr,en}/`: translation catalogs
- `dist/`: intentionally absent; the historical React/Vite build was audited and removed

Before editing a template with duplicates, confirm the active file:

```bash
.venv/bin/python manage.py shell -c "from django.template.loader import get_template; print(get_template('pages/home.html').origin.name)"
```

## Development commands

Initial setup uses the checked-in requirements and public environment template:

```bash
python -m venv .venv
.venv/bin/python -m pip install -r requirements.txt
cp .env.example .env
.venv/bin/python manage.py migrate
```

Run locally over HTTP, not HTTPS:

```bash
.venv/bin/python manage.py runserver
```

Confirmed validation commands:

```bash
.venv/bin/python manage.py check
.venv/bin/python manage.py test
.venv/bin/python manage.py makemigrations --check --dry-run
.venv/bin/python manage.py collectstatic --dry-run --noinput
```

`manage.py test` currently discovers zero tests; a passing result only confirms Django system checks.

Confirmed project commands:

```bash
.venv/bin/python manage.py seed_catalog
.venv/bin/python manage.py optimize_images --dry-run
.venv/bin/python manage.py email_diag
```

- `seed_catalog` is idempotent but writes catalog rows.
- Run `optimize_images --dry-run` first; without it, files under `media/` are replaced.
- `email_diag RECIPIENT` sends an actual email when a real provider is configured.
- `seed_production.py` deletes existing catalog data before recreating it; never run it without explicit authorization.
- Tailwind v4 is compiled locally from `frontend/tailwind.css`: use `npm run dev`, `npm run watch`, or `npm run build`. The canonical output is `static/css/tailwind.css`; run `npm run check:css` to verify Alpine/Django conditional classes. Never hand-edit the generated file. `premium.css` remains separate and loads after Tailwind.

## Coding conventions

- Keep changes inside the owning Django app; shared configuration belongs in `teebusiness_core/settings.py`.
- Views are predominantly function-based. Preserve nearby patterns rather than introducing a new abstraction for a small change.
- Use named URL routes; this project does not use URL namespaces.
- Store money authoritatively in GNF. Use `Decimal` for persisted/order calculations and preserve existing rounding behavior.
- Keep expensive catalog access optimized with `select_related`/`prefetch_related`; `Product.objects.active_listing()` is the shared active-product query.
- User-facing text is a mix of French and English. Preserve the language and `{% trans %}` pattern of the surrounding template; update `locale/` only when the task includes translations.
- No enforced formatter exists. Match the local Python/template style and avoid broad formatting-only diffs.
- Do not silently swallow failures in new core logic. Existing best-effort email, notification, image, and admin-dashboard paths intentionally fail open; preserve that behavior unless asked to change it.

## UI/UX conventions

- `templates/base.html` is the storefront shell; `templates/base_auth.html` is the glassmorphism authentication shell.
- The current visual system uses Inter/Barlow, cream/white surfaces, near-black text, `luxury-*` Tailwind tokens, restrained red branding, rounded panels, and responsive utility classes.
- Reuse `templates/includes/` components such as product cards, heroes, pagination, social links, newsletter, notifications, and WhatsApp UI.
- Alpine powers menus, modals, checkout interactions, toast notifications, and AJAX add-to-cart. Preserve non-JavaScript form fallbacks where they exist.
- `static/css/premium.css` contains active custom storefront styles. The older `static/styles/` files may not be loaded by the active base template; verify references before editing them.
- Check mobile and desktop behavior when changing templates. Preserve keyboard labels, CSRF tokens, semantic form behavior, and the project CSP allow-list.

## Business rules and sensitive areas

- Cart state is stored in the session as `"<product_id>:<size>" -> quantity`. Product/size lines must not be merged across sizes.
- Checkout is the stock authority: validate active products, lock `ProductSize` rows, cap/reject unavailable quantities, calculate shipping/tax server-side, snapshot names/prices into `OrderItem`, decrement stock, and clear the cart atomically.
- Shipping zones and exchange settings come from environment-backed Django settings. Client-side totals are display helpers, never authoritative.
- Product creation, first discount, and restock can synchronously email every active subscriber after transaction commit. Disable `PRODUCT_NOTIFICATIONS_ENABLED` for imports, fixtures, bulk edits, or tests that should not send alerts.
- Product uploads are optimized on new `ProductImage.save()` calls. Preserve fail-open handling and do not reprocess tracked media casually.
- Registration creates an inactive user until OTP verification. Authentication supports customer email first and staff username fallback. Lockout fields and `LoginSecurityLog` are part of the security workflow.
- Django always uses `LoggingEmailBackend`, which wraps console/SMTP/Brevo and writes `EmailHistory`. Avoid duplicate history rows; OTP uses `X-TEE-NoLog` because it journals separately.
- Public order confirmation and invoice routes resolve by hard-to-guess `order_number`; changes to their authorization or URL shape can affect guest checkout.
- Admin actions, exports, subscriber campaigns, stock edits, and order status changes operate on real business data. Do not trigger them as harmless validation.

## Database and migrations

- SQLite is the only configured database. `db.sqlite3` is ignored by policy but is already tracked in this repository; do not modify or commit its contents unless explicitly requested.
- Never edit an applied migration. Change models, create a new migration, inspect it, then run:

```bash
.venv/bin/python manage.py makemigrations
.venv/bin/python manage.py migrate
.venv/bin/python manage.py makemigrations --check --dry-run
```

- `pages` contains historical migrations for a newsletter model that was moved to `communication`; do not recreate the duplicate model.
- Preserve the custom user model and migration dependency ordering. Changing `AUTH_USER_MODEL`, UUID primary keys, order snapshots, or stock fields is high risk.
- Prefer `seed_catalog` for initial catalog data. Treat `seed_production.py` as destructive.

## Static files, media and PDFs

- Edit source assets in `static/`; never hand-edit generated `staticfiles/`.
- `media/` and `logs/django.log` are unexpectedly already tracked despite ignore rules. Preserve them unless the task explicitly covers repository cleanup or asset changes.
- Product files live under `media/products/`. Avoid renames/deletes that orphan `ProductImage` rows.
- Validate static collection with the dry-run command before production-facing changes.
- Admin PDF exports are generated in memory by `teebusiness_core/admin_exports.py`. Customer invoices use `templates/cart/invoice.html` and browser print CSS.
- Logo, favicon, and hero imagery under `static/assets/` are shared across storefront, auth, admin, and emails; check all consumers before replacing them.

## Environment and deployment notes

- `.env` is ignored and may contain secrets. Never print, copy, commit, or place its values in logs, tests, fixtures, or documentation. Keep `.env.example` placeholder-only.
- `settings.py` loads the repository `.env` explicitly. Local development expects `DEBUG=True`, console email, `SITE_URL=http://localhost:8000`, no SSL redirect, and no proxy SSL header.
- Django `runserver` serves HTTP only. Open `http://127.0.0.1:8000/`; `https://...` produces `ERR_SSL_PROTOCOL_ERROR`.
- Production targets PythonAnywhere through WSGI, with `DEBUG=False`, WhiteNoise, HTTPS/proxy settings, trusted origins, secure cookies/HSTS, and Brevo favored because free-tier outbound SMTP may be blocked.
- Product alerts run synchronously. `communication/tasks.py` is a Celery-shaped stub, not an active queue.
- No confirmed production release checklist or automated deployment workflow exists. Do not claim deployment success from local checks.

## Validation checklist

- Review `git diff` and ensure unrelated user changes, tracked data, media, and logs were not altered.
- Run `.venv/bin/python manage.py check`.
- Run relevant tests; also state that the current suite has zero discovered tests if no tests were added.
- For model changes, create/review migrations and run `makemigrations --check --dry-run`.
- For template/UI changes, verify the active template origin and manually exercise affected desktop/mobile, form, CSRF, and no-JS behavior.
- For cart/order changes, test multiple sizes, stock limits, totals, guest/authenticated checkout, stock deduction, and invoice rendering.
- For email/signal changes, use console/locmem email and ensure no real subscribers or providers are contacted.
- For static changes, run `collectstatic --dry-run --noinput`.

## Instructions for AI agents

- Inspect the owning model, view, URL, active template, and related signal/admin code before changing a workflow.
- Keep scope narrow. Do not refactor neighboring code, normalize all languages/styles, or clean tracked artifacts unless requested.
- Preserve existing user changes in the worktree; this repository is commonly dirty.
- Do not modify `.env`, databases, uploaded media, generated assets, or production-facing data without explicit task scope.
- Do not run destructive seeds, real email/SMS sends, bulk admin operations, or image rewrites as validation.
- Add tests for changed business behavior where practical; never describe the empty current suite as meaningful coverage.
- Call out uncertain or unverified behavior explicitly, especially around deployment.
- Before finishing, report exactly what was changed, what was validated, and any checks that could not be performed.
